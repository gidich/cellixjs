CellixJS Shared UI Components

This demonstrates the shared UI component libraries created for the CellixJS monorepo.

@cellix/ui-core Package

Purpose: General reusable UI components for any application

Components: LoadingSpinner, ErrorBoundary, ConfirmDialog

// Usage in React applications:
import { LoadingSpinner, ErrorBoundary, ConfirmDialog } from '@cellix/ui-core';

// Loading spinner
<LoadingSpinner loading={true} tip="Loading data..." />

// Error boundary wrapper
<ErrorBoundary>
  <YourComponent />
</ErrorBoundary>

// Confirmation dialog
<ConfirmDialog 
  open={showDialog}
  title="Delete Item"
  content="Are you sure you want to delete this item?"
  onConfirm={handleDelete}
  onCancel={() => setShowDialog(false)}
/>

@ocom/ui-components Package

Purpose: Owner Community specific UI components

Components: PropertyCard, CommunityHeader, UserStatusBadge

// Usage in Owner Community applications:
import { PropertyCard, CommunityHeader, UserStatusBadge } from '@ocom/ui-components';

// Property card
<PropertyCard
  id="prop-123"
  title="Beautiful Condo"
  address="123 Main St, City, ST"
  rent={2500}
  status="available"
  onClick={(id) => navigateToProperty(id)}
/>

// Community header
<CommunityHeader
  communityName="Sunset Gardens"
  userName="John Doe"
/>

// User status badge
<UserStatusBadge status="owner" />

Build Integration

DevOps Compatible: These packages are built separately from backend packages to avoid React type contamination

# Backend build (unchanged)
npm run build --ws --if-present

# Full build including UI
npm run build

# UI-specific commands  
npm run build:ui
npm run lint:ui
npm run test:ui

Architecture Benefits

  • Isolation: UI packages don't interfere with backend TypeScript compilation
  • Reusability: Components can be shared across multiple applications
  • Type Safety: Full TypeScript support with proper type exports
  • Consistency: Standardized UI patterns across the organization
  • DevOps Friendly: Backend build process remains unaffected
